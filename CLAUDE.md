# Claude Code 开发规范

本文件是 Claude Code 在 `/Users/salty/codeProject/ai` 项目空间内的工作规范。Claude Code 在本目录及其子目录内进行开发、修改、测试、文档维护和提交时，应遵守以下规则。

## 基本工作规范

- 每次完成代码或文档修改后，应提交 git commit。
- commit message 应清晰描述本次修改内容，并在末尾追加对应 model 后缀，例如：`更新用户登录流程 qwen3.6-plus`。
- 当前目录是 Claude Code 的工作目录；创建新项目时，应在 `ai` 目录下创建二级目录，不要直接把新项目文件散落在当前根目录。
- 代码应达到支持生产环境使用的质量标准，保持合理的分层结构、清晰的职责边界和可维护性。
- 请遵守《阿里巴巴 Java 开发手册》相关规范。
- 必须使用可用于生产环境的官方安全的依赖包。

## 注释与日志

- 方法和类上应编写注释，注释内容至少包含：
  - 基本功能描述
  - 创建日期
  - author，统一使用 `sunshengxian`
- 代码中应添加合适的注释，解释关键业务逻辑、复杂分支、边界处理和重要设计决策。
- 代码中应添加合适的日志。
- 日志级别只允许使用 `info` 和 `error`。
- 对处理流程较长的代码，应补充进度类日志，便于线上排查和追踪执行状态。

## 文档与资源

- 大模块或重要功能发生修改时，应同步完善和补充项目根目录的 `README.md`。
- 新增功能涉及的 SQL 脚本，应统一放在项目根目录的 `resources` 文件夹下。
- 允许在 `test` 目录下编写测试用例。

## 依赖与测试约束

- 不允许在项目主体代码中编写 demo 代码或测试代码。
- 不允许使用非官方依赖包。
- 测试代码应放在 `test` 目录或项目既有测试目录结构内。

## 绝对禁止

- 不允许在项目主体中写 demo 和测试代码。
- 不允许使用非官方的依赖包。

## 多 AI 交替开发规范

Claude Code 与 Codex 交替开发 `react-admin-starter` 等项目时，应遵守以下规则。

### 开发前

1. 阅读目标项目的 `todolist.md`，确认"当前进度"和"下一步任务"。
2. 阅读目标项目的 `README.md`，了解当前状态和启动命令。
3. 执行 `git status --short` 确认工作区干净，无未提交的修改。
4. 从"下一步任务"中选择要开发的条目，将其标记从 `[ ]` 改为 `[~]`（表示进行中），并追加完成记录说明。

### 开发中

- 只认领自己负责的 todolist 条目，不修改对方正在进行的条目。
- 遵循项目的工程规范和已有代码风格。

### 开发后

1. 将完成的 todolist 标记从 `[~]` 改为 `[x]`。
2. 更新"当前进度"段落，描述本次完成的范围。
3. 更新"下一步任务"段落，指明后续开发方向。
4. 在"完成记录"末尾追加一条记录，格式：`- YYYY-MM-DD：描述完成内容。`
5. 同步更新 `README.md`（如涉及新增功能、路径变更、启动方式变化等）。
6. 提交 git commit，message 格式：`<描述> `，例如：`实现登录页面逻辑`。

### 标识约定

- Claude Code 的 commit message 末尾追加 `对应模型` 标识。
- Codex 的 commit message 按其模型后缀标识（如 `gpt-5.4`）。
- 双方通过 commit message 即可区分各自的贡献。
